Native Java/Kotlin SSL Disabled AES Key Leaked Native Lib Missing
Critical AES key, IV, and SSL bypass all in plaintext

Internal Infrastructure

Vasundhara's internal Zulip server — exposed

The IP address 103.254.172.165 was found hardcoded in the Universal TV Remote APK pointing to an internal GitLab instance. Investigation of the same host revealed an active Zulip 6.2 team chat server on port 8080 — Vasundhara Infotech's internal communication platform, publicly reachable from the internet.

Critical Zulip 6.2 — 3+ years behind the latest release
Current latest Zulip is version 12.0; this server runs 6.2-4-gaea156e46c (feature level 157). Self-signed TLS certificate, no HSTS, no 2FA, no SSO, no LDAP. Five published CVEs apply, including a HIGH-severity account creation bypass (CVE-2025-31478).

Critical Internal Zulip team chat server exposed to public internet
Vasundhara Infotech's internal Zulip 6.2 server is publicly reachable at 103.254.172.165:8080 — discovered via a hardcoded reference in the Universal TV Remote APK. The server is 3+ years behind the current release (12.0), uses a self-signed TLS certificate (MITM vulnerable), has no 2FA / SSO / LDAP, and is subject to at least 5 unpatched CVEs (including CVE-2025-31478 HIGH 8.2). Realm name "Vasundhara Infotech" and admin email vicky3.vasundhara@gmail.com are publicly visible. Live pen-test confirmed that direct access is blocked by invite-only registration and rate limiting — but the attack surface remains substantial.
Critical GitLab private token & session cookie hardcoded in APK
The Universal TV Remote APK ships with a plaintext GitLab PRIVATE-TOKEN (_nbJ7pT5N4mCY5znnaae) and session cookie (_gitlab_session=86e88493...) granting full read access to project 1602 on the internal GitLab at 103.254.172.165:5000. The server's port is currently closed, but the credentials remain in every installed copy of the app. If the server is restored or migrated, the token grants immediate access.
Critical AES-128 key & IV leaked via Logcat at runtime
The AI Image Generator app encrypts its X-Auth-Header with AES-128-CBC, but the key (3b232c4836fc8671a2fac24e895bac89) and IV (38a7a28a5e66f7f1) are logged via Log.e in RetrofitNetworkManager.java. Anyone reading logcat from an attached device or via a malicious app with log read permission can extract them. Combined with disabled SSL, this enables full authentication bypass and free API abuse.
Critical SSL certificate validation completely disabled
RetrofitClient.java in the AI Image Generator app uses a custom TrustManager that accepts all certificates and an AcceptAllHostnameVerifier. Any proxy or hostile network can MITM the entire traffic. The app's custom encryption interceptor provides only a false sense of security since the AES key is also leaked. HTTP logging is set to Level.BODY in production, leaking image prompts and generated content to Logcat.
Critical Internal infrastructure mapped via APK contents
Across the three apps, the following internal systems are referenced or exposed: internal GitLab (103.254.172.165:5000), internal Zulip (103.254.172.165:8080), DigitalOcean droplet (139.59.92.56 — dead), staging API (fitpodapi.vasundharasolutions.com), local dev IP (10.0.128.8:4000), and primary production APIs at remotecontrol.vasundharaapps.com, pixverseapi.vasundharaapps.com, and api.fitpod.mx. Several use HTTP rather than HTTPS. The shared IP across GitLab and Zulip pivots Vasundhara's entire internal-tooling host into the public attack surface.
High Shelly IoT smart locks control physical gym access
FitPod uses Shelly cloud locks at shelly-213-eu.shelly.cloud to physically unlock gym pods on confirmed bookings. The admin panel has a dedicated /shelly-lock-key route for configuring lock auth keys. If admin credentials or Shelly keys leak via any of the other findings above, an attacker can unlock physical pods remotely — escalating from a digital compromise to physical break-in.
High Staging & dev servers leaked in production config
FitPod's config file contains commented-out references to fitpodapi.vasundharasolutions.com (Vasundhara's internal staging server) and 10.0.128.8:4000 (local dev on Vasundhara's office network). These hint at how the production environment is set up and provide enumeration targets. Vasundhara's vasundharasolutions.com domain suggests separate corporate vs. product infrastructure that may not share the same hardening.
High Native library not bundled — runtime crash or dynamic supply chain
The AI Image Generator declares System.loadLibrary("native-lib") in Secrets.java and Data.java, but no .so file ships in the APK for any ABI. The app must either crash on first launch (UnsatisfiedLinkError) or dynamically download the library at runtime. If the latter, the download URL and signature verification become a hidden supply-chain risk — an attacker who controls the download channel can replace the encryption logic entirely.
Medium 95+ AdMob ad unit IDs hardcoded with shared publisher
All three apps share the same AdMob publisher ID (ca-app-pub-1168261283036318) with 95+ ad unit IDs hardcoded across Java source files. While not directly exploitable, competitors or click-fraud actors can use these to generate fraudulent ad impressions, potentially leading to AdMob account suspension for the entire publisher and revenue loss across all apps.
